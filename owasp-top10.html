OWASP Top 10 - 2021

The most critical web application security risks

Last updated: February 2026

By FixTheVuln Team ¬∑ Peer-reviewed security content ¬∑ Sources: CISA, NVD, OWASP
What is OWASP Top 10? A standard awareness document representing a broad consensus about the most critical security risks to web applications. Updated every few years by the Open Web Application Security Project (OWASP).

Key Takeaways

  • Broken Access Control is the #1 risk in OWASP Top 10 2021, up from #5 in 2017
  • Injection (including SQL injection) dropped to #3 but remains critical
  • Security Misconfiguration moved up to #5 ‚Äî always harden default configs
  • Use parameterized queries, input validation, and output encoding as baseline defenses
  • Server-Side Request Forgery (SSRF) is new to the 2021 list at #10

A01:2021 ‚Äì Broken Access Control

Risk Level: Critical

Description: Users can act outside their intended permissions, accessing unauthorized functionality or data.

Common Vulnerabilities:

‚Ä¢ Bypassing access control checks by modifying URLs, internal application state, or HTML
‚Ä¢ Viewing or editing someone else's account by providing its unique identifier
‚Ä¢ Accessing API with missing access controls for POST, PUT and DELETE
‚Ä¢ Elevation of privilege (acting as a user without being logged in, or as an admin when logged in as a user)

Quick Fixes:

‚Ä¢ Deny by default ‚Äì except for public resources
‚Ä¢ Implement access control mechanisms once and re-use throughout application
‚Ä¢ Enforce record ownership rather than accepting user can create, read, update or delete any record
‚Ä¢ Log access control failures, alert admins when appropriate
‚Ä¢ Rate limit API calls to minimize automated attack harm

// Example: Check user authorization before accessing resource if (!currentUser.hasPermission('edit_post', postId)) { return res.status(403).json({ error: 'Forbidden' }); }

A02:2021 ‚Äì Cryptographic Failures

Risk Level: High

Description: Sensitive data exposed due to lack of encryption or weak cryptographic implementation.

Common Vulnerabilities:

‚Ä¢ Transmitting data in clear text (HTTP, FTP, SMTP)
‚Ä¢ Using old or weak cryptographic algorithms
‚Ä¢ Default or weak crypto keys in use
‚Ä¢ Improper certificate validation
‚Ä¢ Passwords stored without proper hashing

Quick Fixes:

‚Ä¢ Always use TLS/HTTPS for transmitting sensitive data
‚Ä¢ Encrypt all data at rest containing sensitive information
‚Ä¢ Use strong, up-to-date algorithms (AES-256, RSA-2048+)
‚Ä¢ Hash passwords with bcrypt, scrypt, or Argon2
‚Ä¢ Disable caching for responses containing sensitive data

// Example: Proper password hashing with bcrypt const bcrypt = require('bcrypt'); const saltRounds = 10; const hashedPassword = await bcrypt.hash(password, saltRounds);

A03:2021 ‚Äì Injection

Risk Level: Critical

Description: Untrusted data sent to an interpreter as part of a command or query (SQL, NoSQL, OS commands, LDAP).

Common Vulnerabilities:

‚Ä¢ SQL Injection (SQLi)
‚Ä¢ NoSQL Injection
‚Ä¢ OS Command Injection
‚Ä¢ LDAP Injection
‚Ä¢ Expression Language (EL) or OGNL Injection

Quick Fixes:

‚Ä¢ Use parameterized queries (prepared statements)
‚Ä¢ Use ORM frameworks that automatically escape inputs
‚Ä¢ Validate and sanitize all user inputs
‚Ä¢ Use whitelist input validation
‚Ä¢ Escape special characters for specific interpreters

// BAD: Vulnerable to SQL injection query = "SELECT * FROM users WHERE username = '" + username + "'"; // GOOD: Using parameterized query query = "SELECT * FROM users WHERE username = ?"; stmt = connection.prepareStatement(query); stmt.setString(1, username);

A04:2021 ‚Äì Insecure Design

Risk Level: High

Description: Missing or ineffective control design, different from insecure implementation.

Quick Fixes:

‚Ä¢ Establish secure development lifecycle with security professionals
‚Ä¢ Use threat modeling for critical authentication, access control, business logic
‚Ä¢ Integrate security language and controls into user stories
‚Ä¢ Write unit and integration tests to validate all critical flows are resistant to threat model
‚Ä¢ Segregate tier layers on system and network layers

See detailed fix guide ‚Üí

A05:2021 ‚Äì Security Misconfiguration

Risk Level: High

Description: Insecure default configurations, incomplete configurations, open cloud storage, misconfigured HTTP headers.

Common Vulnerabilities:

‚Ä¢ Missing security hardening across application stack
‚Ä¢ Unnecessary features enabled (ports, services, pages, accounts, privileges)
‚Ä¢ Default accounts and passwords still enabled
‚Ä¢ Error handling reveals stack traces or overly informative error messages
‚Ä¢ Latest security features disabled or not configured securely

Quick Fixes:

‚Ä¢ Remove or disable unused features, frameworks, documentation, samples
‚Ä¢ Implement segmented application architecture
‚Ä¢ Send security directives to clients (security headers)
‚Ä¢ Automate verification of configurations in all environments
‚Ä¢ Keep all frameworks, libraries, and dependencies updated

See detailed fix guide ‚Üí

A06:2021 ‚Äì Vulnerable and Outdated Components

Risk Level: High

Description: Using components with known vulnerabilities.

Quick Fixes:

‚Ä¢ Remove unused dependencies, features, files, and documentation
‚Ä¢ Continuously inventory versions of components (client and server-side)
‚Ä¢ Monitor sources like CVE and NVD for vulnerabilities
‚Ä¢ Only obtain components from official sources over secure links
‚Ä¢ Monitor for unmaintained libraries and components

# Check for vulnerable npm packages npm audit # Update vulnerable packages npm audit fix # For severe vulnerabilities npm audit fix --force

A07:2021 ‚Äì Identification and Authentication Failures

Risk Level: Critical

Description: Weaknesses in authentication and session management.

Common Vulnerabilities:

‚Ä¢ Permits automated attacks such as credential stuffing
‚Ä¢ Permits brute force or other automated attacks
‚Ä¢ Permits default, weak, or well-known passwords
‚Ä¢ Uses weak credential recovery processes
‚Ä¢ Exposes session identifiers in the URL
‚Ä¢ Session IDs not properly invalidated

Quick Fixes:

‚Ä¢ Implement multi-factor authentication
‚Ä¢ Do not ship or deploy with default credentials
‚Ä¢ Implement weak password checks
‚Ä¢ Limit or delay failed login attempts (rate limiting)
‚Ä¢ Use server-side, secure, built-in session managers
‚Ä¢ Regenerate session IDs after login

See detailed fix guide ‚Üí

A08:2021 ‚Äì Software and Data Integrity Failures

Risk Level: High

Description: Code and infrastructure that does not protect against integrity violations.

Quick Fixes:

‚Ä¢ Use digital signatures to verify software or data from expected source
‚Ä¢ Ensure libraries and dependencies are from trusted repositories
‚Ä¢ Use software supply chain security tools (OWASP Dependency Check, Snyk)
‚Ä¢ Ensure proper segregation and configuration of CI/CD pipeline
‚Ä¢ Ensure unsigned or unencrypted serialized data not sent to untrusted clients

See detailed fix guide ‚Üí

A09:2021 ‚Äì Security Logging and Monitoring Failures

Risk Level: Medium

Description: Insufficient logging, detection, monitoring, and response capabilities.

Quick Fixes:

‚Ä¢ Log all login, access control, and server-side input validation failures
‚Ä¢ Ensure logs are in format easily consumed by log management solutions
‚Ä¢ Ensure high-value transactions have audit trail with integrity controls
‚Ä¢ Establish effective monitoring and alerting
‚Ä¢ Establish incident response and recovery plan

See detailed fix guide ‚Üí

A10:2021 ‚Äì Server-Side Request Forgery (SSRF)

Risk Level: High

Description: Web application fetches remote resource without validating user-supplied URL.

Quick Fixes:

‚Ä¢ Sanitize and validate all client-supplied input data
‚Ä¢ Enforce URL schema, port, and destination with positive allow list
‚Ä¢ Do not send raw responses to clients
‚Ä¢ Disable HTTP redirections
‚Ä¢ Segment remote resource access functionality in separate networks

// Example: Validate URL before fetching const allowedDomains = ['api.example.com', 'trusted.com']; const url = new URL(userInput); if (!allowedDomains.includes(url.hostname)) { throw new Error('Domain not allowed'); }
