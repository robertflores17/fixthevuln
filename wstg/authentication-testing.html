🔐 WSTG-ATHN: Authentication Testing

OWASP Web Security Testing Guide v4.2

By FixTheVuln Team · Sources: OWASP WSTG v4.2
← OWASP WSTG Guide

Authentication is the gatekeeper of application security. This category tests every aspect of how users prove their identity, from password policies and credential storage to multi-factor authentication and session management during login. Weak authentication is consistently ranked among the top web application risks because it provides direct access to user accounts and sensitive data. Tests in this category cover credential transport security, brute-force protections, password reset flows, and authentication bypass techniques. For security professionals and cert candidates, deep understanding of authentication weaknesses is essential because these vulnerabilities have the most direct business impact.

Detailed Test Cases

WSTG-ATHN-01: Credential Transport Security

Intermediate

This test verifies that user credentials (usernames and passwords) are always transmitted over encrypted channels (HTTPS/TLS). Credentials sent over unencrypted HTTP can be intercepted by anyone on the network path between the user and the server.

Why it matters: Transmitting credentials over unencrypted connections exposes them to network-level interception through packet sniffing, man-in-the-middle attacks, or compromised network infrastructure. This is especially dangerous on public Wi-Fi networks.

What to Check

  • Verify the login form is served over HTTPS and submits to an HTTPS endpoint
  • Check that HTTP-to-HTTPS redirects happen before credentials are submitted
  • Test whether login forms can be loaded over HTTP by directly entering the URL
  • Confirm that API authentication endpoints require HTTPS
  • Verify that HSTS is configured to prevent protocol downgrade attacks
Tools: Burp Suite Wireshark curl testssl.sh Browser Developer Tools
CWE-319 CWE-523

WSTG-ATHN-02: Default Credentials Testing

Intermediate

This test checks whether the application, its infrastructure, or any administrative interfaces still use default credentials that were set during installation. Default credentials are publicly known and are one of the first things attackers try.

Why it matters: Default credentials are published in vendor documentation, security databases, and hacking tools. They represent zero-effort access for attackers and are frequently found in network devices, databases, CMS platforms, and administrative panels.

What to Check

  • Test common default username/password combinations for the identified platform
  • Check for admin accounts with well-known defaults (admin/admin, admin/password)
  • Review vendor documentation for default credentials specific to the software version
  • Test database management interfaces for default database credentials
  • Check for service accounts with default or blank passwords
Tools: Burp Suite Hydra Default Credential databases Nmap scripts Metasploit
CWE-521 CWE-287

WSTG-ATHN-03: Account Lockout Testing

Intermediate

This test evaluates whether the application has adequate brute-force protection through account lockout mechanisms, rate limiting, or progressive delays. It also checks whether the lockout mechanism itself can be abused to deny legitimate users access.

Why it matters: Without lockout mechanisms, attackers can attempt unlimited password guesses until they find the correct one. However, poorly implemented lockouts can be exploited to lock out legitimate users (denial of service) or bypassed through techniques like IP rotation or credential stuffing.

What to Check

  • Determine how many failed attempts trigger an account lockout
  • Test whether lockout can be bypassed by changing IP addresses
  • Check if lockout applies to all authentication methods (web, API, mobile)
  • Verify that lockout notifications are sent to the account owner
  • Test whether an attacker can intentionally lock out other users' accounts
Tools: Burp Suite Intruder Hydra Custom scripts OWASP ZAP
CWE-307

WSTG-ATHN-04: Authentication Bypass

Intermediate

This test attempts to bypass the authentication mechanism entirely through techniques like forced browsing, parameter manipulation, SQL injection in login forms, session token prediction, or exploiting flaws in the authentication logic itself.

Why it matters: Authentication bypass is one of the most severe vulnerabilities because it grants full unauthorized access without needing valid credentials. These flaws often stem from client-side validation, predictable session tokens, or logic errors in the authentication flow.

What to Check

  • Test forced browsing to authenticated pages without logging in first
  • Manipulate authentication parameters (user role, isAdmin, authentication tokens)
  • Test SQL injection in username and password fields
  • Check for authentication bypass through HTTP header manipulation
  • Test if modifying the response from the server can trick the client into granting access
Tools: Burp Suite OWASP ZAP sqlmap Custom scripts
CWE-287 CWE-288

WSTG-ATHN-05: Remember Password Testing

Intermediate

This test examines how the application implements 'remember me' or 'stay logged in' functionality. Insecure implementations may store credentials in cookies, use predictable tokens, or fail to provide adequate expiration and revocation mechanisms.

Why it matters: Insecure 'remember me' implementations can store credentials in plaintext cookies, use predictable token values that attackers can guess, or create long-lived sessions that remain valid even after password changes. These persist the risk of account compromise beyond a single session.

What to Check

  • Examine remember-me cookies for plaintext or reversible credential storage
  • Test whether remember-me tokens are cryptographically random and unpredictable
  • Verify that remember-me tokens are invalidated on password change
  • Check if remember-me functionality has a reasonable expiration time
  • Test whether remember-me tokens are bound to the original client or can be replayed
Tools: Burp Suite Browser Developer Tools Cookie editor extensions
CWE-287 CWE-522

WSTG-ATHN-06: Browser Cache Testing

Intermediate

This test checks whether sensitive data from authenticated pages is stored in the browser cache, history, or back/forward cache. Cached sensitive content can be accessed by subsequent users of a shared computer or recovered through forensic analysis.

Why it matters: On shared or public computers, browser cache can expose sensitive data to the next user. Even on personal devices, cached credentials and sensitive pages can be extracted through malware or physical access to the device.

What to Check

  • Check for Cache-Control and Pragma headers preventing caching of sensitive pages
  • Verify that the back button does not display authenticated content after logout
  • Test whether autocomplete is disabled on sensitive form fields (credentials, SSN, credit card)
  • Check browser history for sensitive URL parameters
  • Verify that no-store directive prevents disk caching of sensitive responses
Tools: Browser Developer Tools Burp Suite curl
CWE-525

WSTG-ATHN-07: Password Policy Testing

Intermediate

This test evaluates the application's password policy for minimum complexity requirements, length restrictions, and whether it prevents commonly used passwords. A weak password policy makes brute-force and dictionary attacks much more likely to succeed.

Why it matters: Weak passwords are the most common cause of account compromise. Applications that allow short, simple, or commonly used passwords make it easy for attackers to guess credentials through automated attacks, especially when combined with account enumeration.

What to Check

  • Test minimum and maximum password length requirements
  • Check if the application enforces complexity rules (uppercase, lowercase, numbers, special characters)
  • Verify that commonly used passwords (password123, qwerty, letmein) are rejected
  • Test if the application checks passwords against known breach databases
  • Confirm that password history prevents reuse of recent passwords
Tools: Burp Suite Manual testing Password policy analyzers
CWE-521

WSTG-ATHN-08: Security Question Testing

Intermediate

This test evaluates the strength of security questions used for account recovery or additional authentication. Weak security questions have answers that are easily guessed, publicly available, or the same for many users.

Why it matters: Security questions are often the weakest link in account recovery. Answers to common questions like 'mother's maiden name' or 'first pet' can be found through social media research, public records, or simple guessing, allowing attackers to bypass authentication entirely.

What to Check

  • Evaluate whether security questions have easily guessable or researchable answers
  • Test if the application allows users to set their own questions with trivial answers
  • Check if security question answers are case-sensitive
  • Verify that brute-force protection applies to security question attempts
  • Test whether security questions can be bypassed through other recovery methods
Tools: Burp Suite Manual analysis Social engineering research
CWE-640

WSTG-ATHN-09: Password Reset Testing

Intermediate

This test examines the password change and reset mechanisms for weaknesses including insecure reset tokens, lack of current password verification during changes, and exploitable password recovery flows that could enable account takeover.

Why it matters: Password reset mechanisms are a prime target for account takeover attacks. Predictable reset tokens, tokens sent via insecure channels, or reset flows that do not properly verify identity can give attackers full control of any user account.

What to Check

  • Test whether password change requires the current password for verification
  • Analyze password reset tokens for randomness and predictability
  • Verify that reset tokens expire within a reasonable timeframe
  • Check if reset tokens are single-use and invalidated after password change
  • Test for host header injection in password reset emails
Tools: Burp Suite Custom scripts Email header analysis tools
CWE-640 CWE-620

WSTG-ATHN-10: Alternative Channel Auth Testing

Intermediate

This test checks whether alternative authentication channels (mobile app, API, legacy interfaces, partner portals) have weaker security controls than the primary web application. Attackers often target the weakest authentication path available.

Why it matters: Organizations often invest heavily in securing their primary web login but overlook alternative channels that access the same backend. A mobile API with no rate limiting or a legacy portal without MFA can undermine the security of the entire system.

What to Check

  • Identify all authentication channels (web, mobile API, legacy systems, partner portals)
  • Compare authentication strength across channels (password policy, MFA, lockout)
  • Test if credentials are shared across channels with different security levels
  • Check if alternative channels bypass security controls enforced on the primary channel
  • Verify that all channels enforce the same session management policies
Tools: Burp Suite Mobile proxy tools API testing tools Postman
CWE-287 CWE-306

Frequently Asked Questions

What are the most critical authentication vulnerabilities in web applications?

The most critical authentication vulnerabilities include default credentials, weak password policies, missing brute-force protections, insecure password reset mechanisms, and authentication bypass through parameter manipulation. These are commonly exploited because they provide direct access to user accounts and often require minimal technical skill to exploit.

How does WSTG-ATHN differ from WSTG-IDNT and WSTG-SESS?

WSTG-IDNT tests how accounts are created and managed, WSTG-ATHN tests how users prove their identity when logging in, and WSTG-SESS tests how that authenticated state is maintained after login. Together, they cover the full lifecycle of user access: creation, verification, and session persistence.

What is credential stuffing and how does it relate to authentication testing?

Credential stuffing uses username/password pairs leaked from data breaches on other sites to attempt logins on the target application. It exploits password reuse across services. Authentication testing evaluates whether the application has defenses against this attack, including rate limiting, account lockout policies, CAPTCHA challenges, and multi-factor authentication.
