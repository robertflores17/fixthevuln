👤 WSTG-IDNT: Identity Management

OWASP Web Security Testing Guide v4.2

By FixTheVuln Team · Sources: OWASP WSTG v4.2
← OWASP WSTG Guide

Identity Management is the foundation upon which authentication and authorization are built. This category examines the processes by which user accounts are created, roles are assigned, and identities are managed throughout their lifecycle. Testing in this area reveals whether an attacker can create unauthorized accounts, enumerate valid usernames, escalate privileges through role manipulation, or exploit weak identity policies. Understanding identity management vulnerabilities is critical for both offensive security testing and secure application design. These tests are highly relevant for certifications like CISSP, which emphasizes Identity and Access Management as a core security domain.

Detailed Test Cases

WSTG-IDNT-01: Role Definitions Testing

Intermediate

This test evaluates how the application defines and enforces user roles. It examines whether roles follow the principle of least privilege, whether role definitions are clear, and whether there are ways to manipulate role assignments to gain elevated access.

Why it matters: Poorly defined roles or overly permissive default role assignments can give users more access than intended. This is a common root cause of privilege escalation vulnerabilities and data breaches involving insider threats.

What to Check

  • Identify all user roles and their associated permissions
  • Verify that roles follow the principle of least privilege
  • Test whether role assignments can be manipulated through client-side parameters
  • Check for overlapping permissions between roles that could create confusion
  • Confirm that administrative roles are properly separated from regular user roles
Tools: Burp Suite Manual testing with multiple accounts Application documentation review
CWE-284 CWE-269

WSTG-IDNT-02: User Registration Testing

Intermediate

This test examines the user registration process for weaknesses including lack of CAPTCHA, acceptance of disposable email addresses, weak identity verification, and the ability to register with duplicate or spoofed identities.

Why it matters: A weak registration process enables attackers to create fraudulent accounts at scale for spam, abuse, or as stepping stones for further attacks. It also affects data integrity if duplicate or fake identities can pollute the user base.

What to Check

  • Test if registration can be automated without CAPTCHA or rate limiting
  • Check if disposable email addresses are accepted for registration
  • Verify that duplicate accounts with the same email or username are rejected
  • Test whether registration parameters can be manipulated to set elevated roles
  • Confirm email verification is required before account activation
Tools: Burp Suite Custom scripts Temporary email services OWASP ZAP
CWE-284

WSTG-IDNT-03: Account Provisioning Testing

Intermediate

This test reviews how accounts are provisioned, modified, and deprovisioned within the application. It checks whether administrative account management follows secure processes and whether former users retain access after deprovisioning.

Why it matters: Improper account provisioning can leave orphaned accounts with active credentials, allow unauthorized privilege changes, or fail to revoke access when employees leave. This is a major concern for compliance frameworks like SOC 2 and ISO 27001.

What to Check

  • Verify that account creation follows an approval workflow for privileged roles
  • Test whether deprovisioned accounts are fully disabled and cannot authenticate
  • Check if account modification actions are logged and auditable
  • Confirm that temporary or test accounts have expiration dates
  • Test whether bulk account provisioning properly validates each account
Tools: Burp Suite Manual process testing Account management API testing
CWE-284 CWE-269

WSTG-IDNT-04: Account Enumeration

Intermediate

This test determines whether an attacker can enumerate valid usernames or email addresses through the application's login, registration, or password recovery functions. Different error messages, response times, or HTTP status codes for valid versus invalid accounts reveal this information.

Why it matters: Account enumeration gives attackers a list of valid targets for credential stuffing, brute-force, and social engineering attacks. Even subtle differences in responses can be used to distinguish valid from invalid accounts at scale.

What to Check

  • Compare error messages for valid versus invalid usernames on the login page
  • Check registration forms for different responses when an existing email is used
  • Test password reset functionality for account enumeration through error messages
  • Measure response time differences between valid and invalid username attempts
  • Look for differences in HTTP response codes or headers between valid and invalid attempts
Tools: Burp Suite Intruder Custom scripts Hydra ffuf
CWE-200 CWE-204

WSTG-IDNT-05: Username Policy Testing

Intermediate

This test evaluates whether the application enforces a strong username policy, including minimum length, allowed characters, and uniqueness requirements. Weak username policies can make accounts easier to guess and harder to manage securely.

Why it matters: Weak username policies allow predictable account names (admin, test, user1) that attackers can easily guess. Combined with weak passwords, predictable usernames make credential attacks trivially easy.

What to Check

  • Test if the application allows very short or single-character usernames
  • Check whether common reserved names (admin, root, test) can be registered by regular users
  • Verify that username format requirements are enforced consistently
  • Test for case sensitivity in username handling (Admin vs admin)
  • Check if usernames can contain special characters that might cause injection issues
Tools: Burp Suite Manual testing Custom registration scripts
CWE-284

Frequently Asked Questions

What is identity management in web application security?

Identity management in web security refers to how applications handle user accounts throughout their lifecycle: registration, provisioning, role assignment, and deactivation. Poor identity management can allow attackers to create fake accounts, enumerate valid users, or gain elevated privileges through flawed role assignment processes.

What is account enumeration and why is it a security risk?

Account enumeration occurs when an application reveals whether a specific username or email exists in the system, typically through different error messages on login, registration, or password reset pages. This is risky because attackers can compile a list of valid accounts to target with brute-force attacks, credential stuffing, or social engineering campaigns.

How do identity management vulnerabilities differ from authentication vulnerabilities?

Identity management focuses on how accounts and roles are created and maintained, while authentication focuses on how users prove their identity. For example, a weak registration process that allows duplicate accounts is an identity management issue, while a weak password policy is an authentication issue. Both are critical, but they target different parts of the access control chain.
