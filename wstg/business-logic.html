📈 WSTG-BUSL: Business Logic Testing

OWASP Web Security Testing Guide v4.2

By FixTheVuln Team · Sources: OWASP WSTG v4.2
← OWASP WSTG Guide

Business logic vulnerabilities are among the most challenging to find and the most impactful when exploited. Unlike injection or configuration flaws that follow known patterns, business logic bugs are unique to each application because they exploit the specific rules and workflows the application implements. This category tests whether business processes can be bypassed, manipulated, or abused, covering everything from data validation and request forgery to file upload restrictions and rate limits. Automated security scanners cannot detect these vulnerabilities because they require understanding the application's intended behavior and testing what happens when that behavior is subverted. For penetration testers and bug bounty hunters, business logic testing is where deep application knowledge pays off with high-severity findings that others miss.

Detailed Test Cases

WSTG-BUSL-01: Business Logic Validation

Advanced

This test checks whether the application properly validates data at the business logic level, beyond basic input validation. It examines whether business rules about data ranges, relationships between fields, and context-specific constraints are properly enforced.

Why it matters: Business logic validation failures allow attackers to submit data that passes technical validation but violates business rules. For example, ordering a negative quantity, setting a future birth date, or submitting a transaction amount that exceeds account limits may all pass input validation but violate business logic.

What to Check

  • Test boundary values for business-specific fields (quantities, amounts, dates)
  • Submit negative values where only positive values should be accepted
  • Test relationships between dependent fields (start date after end date, amounts exceeding limits)
  • Verify that server-side business rules match client-side validation
  • Check if business constraints are enforced consistently across all entry points
Tools: Burp Suite Manual testing Postman Custom scripts
CWE-840

WSTG-BUSL-02: Request Forgery Testing

Advanced

This test checks whether the application relies on client-side data or hidden form fields for business logic decisions that should be enforced server-side. Attackers who can forge or modify requests can manipulate prices, skip validation steps, or change transaction parameters.

Why it matters: Applications that trust client-submitted data for business decisions are vulnerable to request forgery. An attacker can modify prices in hidden form fields, change product IDs, alter quantity values, or manipulate discount codes by intercepting and modifying HTTP requests.

What to Check

  • Modify hidden form fields that control pricing or business logic
  • Change product IDs or SKUs in purchase requests to get different items at wrong prices
  • Manipulate request parameters that control workflow steps or approval states
  • Test if removing or modifying referrer checks allows skipping validation steps
  • Verify that all business-critical values are validated server-side, not just client-side
Tools: Burp Suite OWASP ZAP Postman Browser Developer Tools
CWE-472

WSTG-BUSL-03: Integrity Check Testing

Advanced

This test verifies whether the application implements integrity checks to detect unauthorized modification of data in transit or at rest. It checks for mechanisms like checksums, digital signatures, and tamper-evident logging that protect against data manipulation.

Why it matters: Without integrity checks, attackers can modify data without detection. This includes changing transaction amounts, altering audit logs, modifying configuration files, or tampering with data exchanged between application components. Integrity verification is a core requirement for financial and regulated applications.

What to Check

  • Check if the application uses checksums or HMACs to verify data integrity
  • Test whether modifying data in transit (cookies, hidden fields, API payloads) is detected
  • Verify that audit logs are tamper-evident and modifications are detectable
  • Test if transaction integrity is maintained across multi-step processes
  • Check whether data integrity verification covers all critical data paths
Tools: Burp Suite Custom scripts Manual testing
CWE-354

WSTG-BUSL-04: Process Timing Testing

Advanced

This test checks whether the application is vulnerable to timing-based attacks on business processes. This includes race conditions where concurrent requests can bypass limits, time-of-check-to-time-of-use (TOCTOU) bugs, and timing-based information leakage.

Why it matters: Race conditions and timing attacks can bypass business rules that work correctly for sequential requests. For example, simultaneously submitting two withdrawal requests might allow double-spending if the balance check and deduction are not atomic. These bugs often appear in financial, inventory, and coupon systems.

What to Check

  • Test for race conditions by sending concurrent requests for the same operation
  • Check if time-sensitive operations (OTP validation, coupon redemption) are vulnerable to replay
  • Test for TOCTOU vulnerabilities in multi-step business processes
  • Measure response time differences that might leak information about internal state
  • Verify that rate-limiting and balance checks are atomic operations
Tools: Burp Suite Turbo Intruder Custom multi-threaded scripts race-the-web
CWE-362 CWE-367

WSTG-BUSL-05: Function Limit Testing

Advanced

This test checks whether the application enforces limits on how many times specific functions can be used. This includes rate limiting, usage quotas, one-time-use features (coupons, vouchers, trials), and maximum attempt limits.

Why it matters: Without proper usage limits, attackers can abuse features far beyond their intended use. Single-use coupons can be reused, free trial periods can be restarted indefinitely, and unlimited API calls can be used for data scraping or denial of service.

What to Check

  • Test if single-use features (coupons, voucher codes, trial activations) can be reused
  • Check if rate limiting exists on sensitive operations (login attempts, API calls, downloads)
  • Verify that rate limits are enforced server-side and cannot be bypassed
  • Test if changing session identifiers, IP addresses, or accounts bypasses usage limits
  • Check if free trial limits can be circumvented through account creation
Tools: Burp Suite Intruder Custom scripts Postman
CWE-770 CWE-799

WSTG-BUSL-06: Workflow Circumvention

Advanced

This test checks whether the application enforces the intended sequence of steps in multi-step processes. Attackers may attempt to skip, reorder, or repeat steps in workflows like registration, checkout, or approval processes to achieve unintended outcomes.

Why it matters: Workflow circumvention can allow attackers to bypass critical security or business steps. Skipping payment verification in a checkout process, bypassing email verification in registration, or jumping directly to the approval step in a review process can have severe business consequences.

What to Check

  • Test if steps in multi-step processes can be skipped by directly requesting later steps
  • Check if workflow steps can be completed out of order
  • Verify that completing a workflow step requires all previous steps to be genuinely completed
  • Test if repeating earlier steps after completing later steps causes inconsistencies
  • Check whether canceling and restarting workflows preserves or resets state correctly
Tools: Burp Suite Manual testing Custom scripts
CWE-841

WSTG-BUSL-07: Application Misuse Testing

Advanced

This test evaluates whether the application has defenses against abuse of legitimate functionality. This includes detecting and blocking automated bot activity, identifying suspicious usage patterns, and implementing safeguards against feature abuse at scale.

Why it matters: Applications without abuse detection allow attackers to exploit legitimate features for malicious purposes. Scraping user data, creating fake accounts at scale, manipulating review systems, or exploiting referral programs are all examples of application misuse that require proactive detection and prevention.

What to Check

  • Test if automated interactions are detected and blocked (CAPTCHAs, bot detection)
  • Check if suspicious usage patterns trigger alerts or account restrictions
  • Verify that abuse-prevention measures cannot be easily bypassed
  • Test if the application logs and monitors for abnormal usage patterns
  • Check if there are mechanisms to recover from detected abuse scenarios
Tools: Burp Suite Custom automation scripts Selenium Puppeteer
CWE-799 CWE-837

WSTG-BUSL-08: Unexpected File Type Upload

Advanced

This test checks whether the application properly restricts file upload functionality to only accept intended file types. Accepting unexpected file types can lead to web shell uploads, stored XSS through HTML/SVG files, or server-side processing vulnerabilities through malicious documents.

Why it matters: Unrestricted file upload is a severe vulnerability that can lead to remote code execution if an attacker uploads a web shell, cross-site scripting if HTML or SVG files are served from the same domain, or server compromise through malicious files that exploit document processing libraries.

What to Check

  • Test if the application validates file types by extension, MIME type, and file content (magic bytes)
  • Try uploading executable files with changed extensions (.php.jpg, .asp;.jpg)
  • Test if MIME type validation can be bypassed by modifying Content-Type headers
  • Upload HTML and SVG files that contain JavaScript to test for stored XSS
  • Check if uploaded files are served from a separate domain to prevent same-origin attacks
Tools: Burp Suite Custom file payloads OWASP ZAP
CWE-434

WSTG-BUSL-09: Malicious File Upload

Advanced

This test goes beyond file type restrictions to check whether the application is vulnerable to malicious content within uploaded files. This includes web shells, documents with embedded macros, polyglot files, and specially crafted files that exploit processing vulnerabilities.

Why it matters: Even when file type restrictions are in place, malicious content within allowed file types can compromise the system. ZIP bombs can cause denial of service, images with embedded PHP can become web shells on misconfigured servers, and office documents with macros can compromise users who download them.

What to Check

  • Upload web shells disguised as allowed file types (PHP in JPEG, JSP in PDF)
  • Test for ZIP bomb or decompression bomb handling
  • Upload files with embedded macros or active content
  • Test for file processing vulnerabilities (ImageTragick, LibreOffice CVEs)
  • Check if antivirus scanning is performed on uploaded files before they are accessible
Tools: Burp Suite Custom file payloads eicar test file fuxploider
CWE-434 CWE-509

Frequently Asked Questions

What are business logic vulnerabilities and why are they hard to find?

Business logic vulnerabilities are flaws in the application's intended workflow that allow users to achieve outcomes the developers did not anticipate. They are hard to find because they require understanding the application's business rules and testing edge cases that automated scanners cannot identify. Examples include skipping required steps in a checkout process, manipulating quantities to get negative prices, or reusing single-use coupon codes.

Can automated scanners detect business logic flaws?

No, automated scanners generally cannot detect business logic flaws because these vulnerabilities require understanding the intended behavior of the application. A scanner cannot know that skipping step 3 of a 5-step process should not be allowed, or that a coupon should only work once. Manual testing by someone who understands the application's business rules is essential for finding these issues.

What is the relationship between business logic testing and threat modeling?

Threat modeling identifies potential business logic abuse scenarios before testing begins by analyzing how the application's features and workflows could be manipulated. It asks questions like 'What if a user submits step 4 before step 2?' or 'What if the quantity is negative?' This systematic approach ensures business logic testing is comprehensive rather than relying on ad-hoc exploration.
